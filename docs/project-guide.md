# 프로젝트 운영 가이드

새 프로젝트를 시작할 때 이 문서를 복사해서 `docs/`에 넣고, 아래 순서대로 세팅한다.

---

## 1. 프로젝트 정보 (채우기)

아래 항목을 프로젝트에 맞게 채운다. 이 내용은 `CLAUDE.md`에 들어간다.

```
프로젝트명:
한 줄 설명:
기술 스택:
실행 방법:
```

**구조 트리** — 핵심 파일만 정리 (venv, node_modules 등 제외):

```
project/
├── CLAUDE.md
├── src/
├── docs/
│   ├── todo.md
│   └── changelog.md
└── ...
```

**문서 참조 맵** — 어떤 정보가 어디에 있는지:

```
할일/진행 상황 → docs/todo.md
작업 이력     → docs/changelog.md
설계 문서     → docs/plan.md (있는 경우)
시크릿 위치   → docs/secrets.md (있는 경우)
```

---

## 2. 문서 세팅

아래 양식대로 파일 3개를 만든다.

### CLAUDE.md (프로젝트 루트)

```markdown
# [프로젝트명]

[한 줄 설명]

## 프로젝트 구조

[1절에서 작성한 구조 트리]

## 실행 방법

[로컬 실행 명령어, 환경변수 등]

## 작업 시 참고

[1절에서 작성한 문서 참조 맵]

## 작업 규칙

- 코드 작업 후 반드시 docs/todo.md와 docs/changelog.md를 함께 업데이트
- 프로젝트 구조가 바뀌면 이 파일(CLAUDE.md)도 업데이트
- 모든 작업은 "계획 → 검토 → 개선 → 검증 → 확정 → 구현" 프로세스를 따른다
```

### docs/todo.md

```markdown
# Todo

## 진행 중

(없음)

## 할 일 (Backlog)

- [ ] 항목

## 완료

### [작업명] — YYYY-MM-DD
- [x] 항목
```

### docs/changelog.md

```markdown
# Changelog

## YYYY-MM-DD
- 작업 내용
```

## 3. 작업 프로세스

모든 작업은 아래 루프를 따른다. 한 번에 완벽할 필요 없이, 루프를 돌면서 나아진다.

```
계획 → 검토 → 개선 → 검증 → 확정 → 구현
 ↑                              ↓
 └──────── 다음 작업 ←──────────┘
```

| 단계 | 하는 일 |
|------|---------|
| **계획** | 작업 범위, 접근 방식, 영향 범위를 정리한다 |
| **검토** | 빠진 것은 없는지, 더 나은 방법은 없는지 스스로 점검한다 |
| **개선** | 검토 결과를 반영하여 계획을 수정한다 |
| **검증** | 최종 계획이 요구사항을 충족하는지 확인한다 |
| **확정** | 사용자에게 보여주고 승인을 받는다 |
| **구현** | 코드 작업 + 문서 업데이트를 동시에 한다 |

핵심: **확정 전에는 코드를 건드리지 않는다.** 확정 후에만 구현에 들어간다.

---

## 4. 작업 규칙

이 규칙은 프로젝트 종류와 무관하게 항상 적용한다.

1. **문서는 코드와 함께 움직인다**
   - 작업 시작 → `todo.md`에서 "진행 중"으로 이동
   - 작업 완료 → `todo.md` 체크 + `changelog.md`에 이력 추가
   - 구조 변경 → `CLAUDE.md` 업데이트

2. **프로세스를 건너뛰지 않는다**
   - 급해도 계획 → 확정 → 구현 순서를 지킨다
   - 작은 작업이라도 최소한 "이렇게 하겠다" 한 줄은 정리하고 시작한다

3. **스스로 진화한다**
   - 매 루프에서 이전보다 나은 결과를 낸다
   - 검토 단계에서 자신의 계획을 비판적으로 본다
   - "이게 최선인가?"를 습관적으로 묻는다
